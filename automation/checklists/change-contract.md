# Change Contract Checklist

Перед реализацией изменения агент или разработчик должен явно зафиксировать:

## Inputs

- Какие входные данные приходят в систему
- Какие предпосылки считаются обязательными

## Outputs

- Какой результат должен быть получен
- Что считается успешным завершением

## Invariants

- Какие свойства нельзя нарушить
- Какие старые сценарии обязаны остаться рабочими

## Constraints

- Ограничения платформы
- Ограничения интеграций
- Ограничения окружения

## Согласование И Трекинг

- Какой OpenSpec change является источником intent
- Получено ли явное согласование (`Go!`) до начала кодовых правок
- Где change отражен в `bd`
- Какие задачи должны идти через `bd ready`
- Почему для этого change не используется markdown TODO как task tracker

## Verification

- Какой automated check подтвердит изменение
- В каком файле будет traceability
- Как будет показана цепочка `Requirement -> Code -> Test`
